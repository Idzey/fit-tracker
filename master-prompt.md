## РОЛЬ И КОНТЕКСТ

Ты выступаешь одновременно как **Staff Software Engineer, Product Manager, Solution Architect и Tech Lead**. Твоя задача — спроектировать мобильное SaaS-приложение для персональных фитнес-тренеров.

**Главная цель:** проект уровня портфолио, который выглядит как реальный коммерческий продукт и демонстрирует навыки **React Native разработчика уровня Mid**.

**Калибровка сложности (важно):** делай решения production-реалистичными, но НЕ переусложняй. Если какая-то тема (например, мульти-региональное масштабирование, сложный биллинг, аудит на уровне enterprise) выходит за рамки того, что Mid-разработчик может уверенно защитить на собеседовании, — упрости её, отметь как «упрощено для портфолио» и опиши, как это решалось бы в реальном проде. Я должен суметь объяснить каждое архитектурное решение как своё собственное.

## ТРЕБОВАНИЯ К ФОРМАТУ АРТЕФАКТОВ

Не описывай словами то, что можно показать кодом или диаграммой. Конкретно:

- **Схема БД** → готовый `schema.prisma` (полный код) + **ER-диаграмма в Mermaid**.
- **API** → спецификация **OpenAPI 3.1 в YAML** (маршруты, схемы, примеры запросов/ответов, коды ошибок).
- **Архитектурные диаграммы** (системная, realtime, загрузка файлов, аутентификация) → **Mermaid**.
- **DTO, права доступа (permissions matrix), тарифные планы** → **таблицы Markdown**.
- **Структура папок** (frontend и backend) → дерево в блоке кода.
- **User journeys и спринты** → структурированные списки/таблицы.

## РЕЖИМ РАБОТЫ: ПОЭТАПНО, С ПОДТВЕРЖДЕНИЕМ

⚠️ **Не выдавай всё сразу.** Полный объём — это ~13 крупных разделов; вывод их одним сообщением приведёт к поверхностности. Работай так:

1. **Шаг 0 — План.** Сначала выдай только: (а) краткое оглавление всего документа, (б) разбивку проекта на **спринты** (см. ниже), (в) список открытых вопросов/допущений, которые ты делаешь. Затем остановись и жди моего «продолжай».
2. После моего подтверждения выдавай по **одному разделу за раз** (или по одному спринту), в глубину, с рабочими артефактами.
3. В конце каждого раздела — короткий чек-лист «что готово / что осталось» и вопрос, переходить ли к следующему.

## РАЗБИВКА НА СПРИНТЫ

Раздели всю разработку на спринты (примерно 2-недельные), как в реальной команде. Для каждого спринта укажи:

- **Цель спринта** (одно предложение)
- **Scope** — что входит (epics / user stories)
- **Артефакты/deliverables** на выходе спринта
- **Definition of Done**
- **Зависимости** от предыдущих спринтов
- **Риски и trade-offs**

Предложи логичную последовательность, например: Sprint 0 (инфраструктура, Docker, CI, схема БД, auth-скелет) → дальше по вертикальным срезам фич (auth → клиенты → тренировки → прогресс → realtime → подписки → уведомления → аналитика → полировка). Финальный спринт — стабилизация, тесты, observability, подготовка к демо. Явно отметь, что входит в **MVP**, а что — в **post-MVP / roadmap**.

## СОДЕРЖАНИЕ ПРОДУКТА

### Обзор продукта

Две роли:

**Тренер:** создаёт клиентов и управляет ими; назначает программы тренировок; создаёт шаблоны тренировок; отслеживает прогресс; просматривает загруженные фото прогресса; получает уведомления о завершённых тренировках; управляет подпиской; смотрит аналитику и активность клиентов.

**Клиент:** получает назначенные программы; отмечает упражнения и тренировки выполненными; загружает фото прогресса; отслеживает личный прогресс; получает push-уведомления и напоминания; смотрит историю тренировок.

### Технологический стек

**Frontend:** React Native, Expo, TypeScript, Expo Router, NativeWind, React Native Reusables, TanStack Query, Orval (генерация из OpenAPI), Zustand, React Hook Form, Zod.

**Backend:** Node.js, Fastify, TypeScript, Passport, Prisma, PostgreSQL, Docker Compose.

Дополнительные библиотеки — только с обоснованием.

### 1. Архитектура продукта

Роли и права; разбивка по фичам; **стратегия монетизации**; тарифные планы; user journeys; границы MVP; roadmap.

> **По монетизации обязательно раскрой:** как обрабатываются платежи и подписки на мобайле. Сравни **Stripe** (web/billing) и **RevenueCat / нативные in-app purchase** (App Store / Google Play) — с учётом комиссий сторов и правил публикации. Объясни разницу между web-подпиской и in-app purchase и какой путь выбираешь для этого продукта и почему.

### 2. Системная архитектура

Frontend, backend, БД, API, realtime, загрузка файлов, уведомления — каждая с диаграммой Mermaid.

### 3. Проектирование БД

Полный `schema.prisma` + ER-диаграмма. Сущности: users, trainer_profiles, client_profiles, workouts, workout_templates, workout_days, exercises, exercise_logs, workout_logs, progress_photos, subscriptions, notifications. Опиши связи и обоснование каждой.

### 4. Backend-модули

auth, users, trainers, clients, workouts, exercises, progress, subscriptions, uploads, notifications, analytics. Для каждого: зоны ответственности, эндпоинты, DTO (таблица), валидация (Zod-схемы), права доступа (permissions matrix).

### 5. Аутентификация

email/пароль; JWT access token; **ротация refresh-токенов**; role-based авторизация. Диаграмма потока auth + обработка кражи/реюза refresh-токена.

### 6. Realtime

Обновления для: завершения тренировок, дашборда тренера, отслеживания прогресса, уведомлений.

> **Не просто «выбери технологию».** Сравни варианты — **WebSocket vs SSE vs polling**, и **своё решение на Fastify vs managed (Pusher/Ably)** — таблицей trade-offs. Учти специфику Expo/React Native: фоновый режим приложения, переподключения, потеря сети, дублирование событий. Затем обоснуй выбор для Mid-уровня.

### 7. Хранилище (фото)

Провайдер хранилища; стратегия доступа (signed URLs); модель безопасности; оптимизация изображений (ресайз, форматы, thumbnails).

### 8. Уведомления

Напоминания о тренировках; уведомления о завершённых тренировках; алерты тренеру. Опиши пуш-инфраструктуру (Expo Push / FCM-APNs) и планировщик напоминаний.

### 9. Экраны мобильного приложения

**Тренер:** dashboard, clients, client details, workout builder, analytics, settings.
**Клиент:** home, workouts, workout details, progress, photos, profile.

Дай карту навигации (Expo Router) и список состояний экрана.

### 10. Frontend-архитектура

Структура папок (дерево); feature-based архитектура; стратегия TanStack Query (ключи, инвалидация, кэш); структура стора Zustand; workflow генерации Orval; архитектура форм RHF + Zod.

### 11. API Design

OpenAPI 3.1 YAML: маршруты, примеры запросов/ответов, пагинация, фильтрация, единый формат ошибок.

### 12. Безопасность

Авторизация; rate limiting; валидация; хеширование паролей (argon2/bcrypt — обоснуй); защита загрузки файлов (тип, размер, скан); аудит-логирование.

### 13. DevOps

Docker Compose; workflow локальной разработки; CI/CD pipeline (GitHub Actions); управление окружениями (env, секреты).

## НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ (обязательно)

Отдельным блоком раскрой:

- **Тестовая стратегия:** что покрывается unit / integration / e2e; инструменты (Jest, React Native Testing Library, Detox/Maestro, supertest); что тестировать в первую очередь при ограниченном времени.
- **Offline-first (критично для мобайла):** поведение при потере сети, очередь действий (отметка тренировки офлайн), синхронизация, persist кэша TanStack Query.
- **Оптимистичные апдейты** в TanStack Query: где применяются и как откатываются.
- **Error & loading states:** единый подход к ошибкам/загрузке/пустым состояниям на фронте.
- **Observability:** логирование (структурные логи), метрики, error tracking (Sentry), health-checks.
- **Масштаб:** оцени порядок нагрузки (сколько тренеров/клиентов/запросов) и где появятся узкие места — но без преждевременной оптимизации.
